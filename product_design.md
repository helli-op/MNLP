\section*{1. Контекст проекта}
\subsection*{1.1 Бизнес-задача}
Каждая компания обладает уникальной корпоративной культурой, культурой написания кода и организации рабочего процесса, из-за чего даже у опытных IT-работников возникают сложности при смене места работы. При этом у руководителя и/или бадди (которого очень редко выделяют) нет ресурса рабочего времени на полноценный онбординг нового сотрудника. Из-за этого эффективность работы падает: новый сотрудник долго "втягивается" в рабочий процесс. Одним из доступных решений этой проблемы является создание вики-страничек с пояснениями и туториалами. Но по мере развития команд страниц становится очень много, стуктурность теряется из-за редактирования многими сотрудниками и в целом, на личном опыте, разобраться в таких вики с нуля на грани невозможного. Оптимальным решением такой проблемы может выступать "RAG-ассистент для корпоративного обучения". Он самостоятельно будет искать нужную информацию из имеющеся в системе и помогать новым сотрудникам адаптироваться. Все вышеописанное касается также и давно работающих в компании сотрудников, при условии освоения новой системы/адаптации к изменениям в процессах.

Этот продукт можно назвать успешным, если мы увидим активное его использование со стороны сотрудников, а также если загруженность руководителей и команд поддержки орг вопросами упадет.

\subsection*{1.2 Целевая аудитория и пользователи}
Продукт рассчитан на внутрениих, в частности новых сотрудников компании. В качестве MVP будет сделан веб-интерфейс, однако в дальнейшем можно обернуть проект, например, в бот. Нагрузка на ассистент не должна быть большой, так как сотрудник будет его использовать в основном в процессе онбординга, но также сможет возвращаться к нему в любое время. Сотрудники выходят в разное время, поэтому колоссальной нагрузки не ожидается. Однако, набор стажеров происходит волнами, и в это время ожидается, что число пользователей резко возрастет, но все еще окажется в зоне допустимых значений.

\subsection*{1.3 Ограничения и допущения}
Ограничения:
- Бот отвечает только в рамках корпоративного обучения.
- Не выполняет действия от имени пользователя.

Мы делаем следующие допущения:
- Документы доступны в цифровом виде.
- Есть доступ к внутренней базе знаний, в которой хранятся все документы.
- Допускается использование внешней LLM.
- Все документы для продукта в базе знаний в формате PDF.

\section*{2. Архитектура решения}
\subsection*{2.1 Общая схема}
Пользователь → Chat UI → API Gateway → Core Service → Retriever → Векторная БД → LLM → Пост-обработка → Ответ
Алгоритм процесса:

1. Пользователь инициирует запрос через интерфейс чата

2. В чате отображается запрос, он же отправляется в бэкенд и визуализирует ответ

3. API Gateway маршрутизирует запросы, ограничивает частоту запросов

4. Core Service - основной сервис: передает запрос в Retiever, получает найденные фрагменты, формирует промт, управляет потоком данных

5. Retriever векторизует запрос пользователя, выполняет поиск по БД, возвращает топ n наиболее релевантных чанков с метаданными.

6. Векторная БД хранит векторные эмбеддинги документов, обеспечивает поиск, поддерживает фильтрацию

7. LLM - наша модель, которая генерирует ответ на основе промпта и контекста

8. Пост-обработка фильтрует нежелательный контекнт, форматирует ответ

\subsection*{2.2 Хранилище знаний / документооборот}
Мы сделали допущение, что все документы, касающиеся онбординга, хранятся во внутренней базе знаний к которой у нас есть доступ, так что все документы будут доставаться оттуда. При обновлении документов они также обновляются в базе знаний. Информация на онбординге не меняется с какой-то периодичностью, так что данные, полученные из базы знаний, можно обновлять вручную при необходимости.

Шаги предобработки:
- Снятие текстового слоя при необходимости (OCR)
- Очистка текста
- Разбиение на чанки

Шаги индексации:
- Векторизация с помощью embedding-модели
- Метрика cosine similarity для вычисления расстояния
- Добавление метаданных при необходимости

\subsection*{2.3 Интеграции и интерфейсы}
В рамках текущей реализации система интегрируется с веб-интерфейсом через REST API. 
Чат будет выглядеть как стандартный интерфейс для использования ассистента, будут кнопки для оценки ответа, ссылки на источники и возможность открыть документ

\subsection*{2.4 Инфраструктура и развертывание}
1. Контейнеризация: Docker для упаковки всех компонентов (Frontend, Core Service, Retriever и т. д.) в изолированные контейнеры с предсказуемой средой выполнения.

2. Оркестрация: Kubernetes для:

-автоматического развёртывания и обновления контейнеров

-балансировки нагрузки между репликами сервисов

-самовосстановления (перезапуск упавших контейнеров)

-горизонтального масштабирования (HPA — Horizontal Pod Autoscaler)

3. Serverless‑компоненты для фоновых задач:

-индексации новых документов;

-переиндексации базы знаний;

-обработки пакетных запросов;

4. Балансировка нагрузки для распределения трафика между инстансами API Gateway.

5. Кэширование: Redis для кэширования частых запросов и сессий.

\section*{3. Данные и качество знаний}
Документы не содержат чувствительные данные, так что для реализации продукта можно использовать внешние LLM-модели
\subsection*{3.1 Сбор и предобработка данных}
Основной источник документов - база знаний, формат документов - только PDF (указали в допущении). Шаги предобработки: очистка, удаление дублей, разбиение на чанки, добавление метаданных при необходимости.
\subsection*{3.2 Векторизация и индексирование}
Embedding-модель должна быть multilingual, метрика - cosine similarity. 
\subsection*{3.3 Метрики качества знаний}
Документы содержат полную информацию, необходимую для онбординга. База обновляется при внесении изменений вручную. Дубликатов и противоречивых данных нет. Проверка будет проходить с помощью ручной ревизии.

\section*{4. Модель и генерация}
\subsection*{4.1 Выбор LLM и промптинг}
В качестве основной модели была выбрана модель DeepSeek-V3.2, так как она хорошо поддерживает русский язык, быстро работает и показывает хорошие метрики на бенчмарках. В промпт-шаблоне будет заложен системный промпт, в котором будет указана роль ассистента, инструкция, как работать с данными и формат ответа, а также контроль галлюцинаций и прочие ограничения ("Не выдумывай", "Отвечай только по контексту").

Контекстное окно выбранной модели - 128000 токенов. Ограничение по ответу модели разумно поставить в районе 10 процентов от контекста модели, чтобы избежать галлюцинаций. Было выбрано ограничение в 10000 токенов. Также в параметрах стоит указать температуру 0, так как задача ассистента в поиске информации в документах, а не генерации новой.

\subsection*{4.2 Контроль качества ответов}
Для контроля качества ответов предусмотрены кнопки удовлетворенности и неудовлетворенности ответами в интерфейсе, благодаря чему можно собирать информацию и анализировать с помощью различных метрик, например, F1-score.

\subsection*{4.3 Обучение/дообучение}
На данном этапе fine-tuning не предусмотрен, так как работа ассистента заключается в основном в использовании RAG-системы и ассистент не должен отвечать на вопросы, не связанные с онбордингом. Однако, стоит оценить пользовательский опыт и, при необходимости, дообучить модель.

\section*{5. UX / пользовательский опыт}
\subsection*{5.1 Сценарии взаимодействия}
1. Приветствие: пользователь запускает чат (нажимает "Начать"), ему автоматически отправляется приветственное сообщение.

2. Поиск по знаниям: пользователь задает прямой вопрос (например, "Как получить доступ к чувствительным данным?"). Система обрабатывает запрос, ищет релевантные фрагменты в базе данных, генерирует ответ с использованием LLM.

3. Уточняющий вопрос: пользователь задает доп вопрос к предыдущему ответу. Система учитывает контекст предыдущего диалога, ищет информацию, связанную с текущей темой и дает развернутый ответ.

4. Запрос принят на рассмотрение: как только бот получил вопрос, он должен ответить, что запрос принят и ответ уже ищется.

Исключительные сценарии:

1. Нет релевантного ответа в базе знаний: система не находит подходящий фрагмент в векторной БД, отвечает шаблоном "Не удалось найти информацию по вашему запросу" и предлагает альтернативные действия: переформулировать вопрос, предлагает походие темы

2. Запрос не понят: система отвечает шаблоном: «Уточни, пожалуйста, вопрос»

3. Технический сбой: ошибка подключения к базе данных/таймаут/внутренняя ошибка обработки. Система отображает сообщение-предупреждение, логирует ошибку и автоматически оповещает команду поддержки.

4. Система перегружена: в таком случае бот должен отправлять сотруднику оповещение об этом и просьбу подождать.

\subsection*{5.2 Диалоговая логика}
Для качественной помощи нужно использовать multi‑turn логику, чтобы иметь возможность отвечать на уточняющие вопросы. Для экономии ресурсов будем ичпользовать окно истории: бот будет помнить последние сообщения за день: так как вопросы задаются в рамках одной сессии в течение рабочего дня больше контекста излишне. Для хранения можно использовать Redis. 

Бот должен отвечать вежливо, но дружелюбно и на "ты", так как он будет использоваться в айти компаниях. Деловой тон в меру должен присутствовать, поэтому эмодзи, неформальные реакции и личностные тезисы ("Ты большой молодец, что интересуешься" и тп) не нужны. Язык ответов - русский.
\subsection*{5.3 Метрики UX}
На запрос бот должен отвечать стандартным шаблоном ("Твой вопрос услышан, работаю над поиском ответа"/ "К сожалению, бот перегружен, отвечу на вопрос как только смогу") практически мгновенно (установим ограничение до минуты), в качестве контролирующей метрики будем смотреть на долю сессиий с ожиданием > минуты.

Затем важно отслеживать скорость основного ответа. Будем смотреть на среднее, медиану и 90й персентиль, чтобы видеть массовые аномалии, если такие есть.

Важно быть уверенными, что база знаний бота достаточно полноценная, чтобы закрывать вопросы сотрудников. Для контроля этого будем наблюдать за долей сессий, в которых бот не нашел релевантных ответов (отдельный сценарий у нас)

Будем собирать NPS и следить за ним (для начала поставим цель на уровне 3.5). Раз в 2-3 месяца будем собирать развернутый CSAT.
\section*{6. Безопасность, соответствие и этика}
Здесь не требуется особых замечаний, так как бот будет работать только с документацией, связанной с онбордингом в различные системы. Он не будет взаимодействовать с чувствительными данными, ему не нужна персональная информация о сотруднике и тд. 


\section*{7. План внедрения и эксплуатации}
\subsection*{7.1 Этапы проекта}
- Фаза 0 - аналитика (1–2 месяца): аудит данных, проектирование архитектуры, определение KPI.
- Фаза 1 - MVP (2–3 месяца): индексация части базы, базовый чат, ограниченный департамент.
- Фаза 2 - расширение: персонализация, интеграции, улучшенный UX.
- Фаза 3 - масштабирование: оптимизация, мониторинг, SLA.

\subsection*{7.2 Поддержка и эксплуатация}
За поддержку отвечает команда ассистента, например, ML-инженеры и Backend-разработчики. Мониторинг будет проходить с помощью логирования ошибок, SLA, алертов и дэшбордов. Метрики эксплуатации: uptime, latency, cost per session.

\section*{8. Риски и допущения}
1. Модель дает неточные или вообще ложные ответы. (средняя вероятность) Мы будем противостоять этому, так как внедряем RAG-архитектуру. Можно добавить механизм дополнительной обратной связи (полезно/не полезно)

2. База знаний устаревает или недостаточно. (высокая вероятность) Мы сможем это мониторить по UX метрикам выше, но вообще, конечно система будет требовать регулярной довыгрузки данных и их актуализации.

3. Стоимость. (низкая вероятность) Содержать LLM не бесплатно. Мы добьемся удешевления засчет оптимизации запросов и не хранения большого количества контекста сессий.

4. Сопротивление пользователей. (высокая вероятность) Сотрудники могут относиться с недоверием к продукту. Для противодействия этому можно сделать открытую презентацию системы + дополнительное оповещение в корпоративном мессенджере.

5. Некорректная обработка многоходовых диалогов. (средняя вероятность). Для борьбы будем четко определять границы сессии, добавим кнопку "новый диалог" для сброса контекста.

\section*{9.Бюджет и ресурсы}

Для проекта понадобится 2 DL инженера (для построения и настройки модели, оптимизации поиска, предобработки текста), фронтенд и бэкенд разработчик (развертывание и оркестрация, мониторинг и алертинг, управление ресурсами), аналитик (для формирования продуктовых метрик, настройки мониторинга, проведения UX исследований), UX-дизайнер, менеджер продукта.

На данном этапе сложно рассчитать затраты, так как они зависят от размера компаний, от ресурсов, которыми обладает компания на данный момент и тд)
